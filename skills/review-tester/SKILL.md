# Эксперт по качеству тестов (review-tester)

## Задача

Оценить качество тестов (юнит‑ и интеграционных) в pull request или при саморецензии.  
Выявить test‑smells, проверить покрытие изменённого кода (≥80%), при необходимости сгенерировать заглушки тестов для непокрытых методов.  
Вывод — только JSON.

## Известные проблемы (Test Smells)

### ВЫСОКАЯ важность (HIGH) — must fix перед merge

| Test Smell | Симптом | Рекомендация |
|------------|---------|----------------|
| **Отсутствие assertions** | В тесте есть Act, но нет Assert | Добавить проверку бизнес‑результата |
| **Тесты зависят друг от друга** | Общее состояние, порядок выполнения важен | Изолировать тесты, использовать свежие фикстуры |
| **Недомокание (under‑mocking)** | Реальные БД/API вызовы в unit‑тестах | Мокать внешние зависимости |
| **Mystery guest** | Внешний ресурс без явного setup | Явно настраивать тестовые данные |
| **Несоответствие мока интерфейсу** | Мок использует не тот интерфейс, что реализация | Синхронизировать с актуальным интерфейсом |
| **Flaky тест** | Проходит/падает недетерминированно | Убрать случайности, гонки, общее mutable состояние |
| **Async void / потерянный await** | Асинхронный тест без `await` или с `.Result` | Всегда использовать `await`, тест возвращает `Task` |
| **Хардкод секретов** | Пароли, ключи, токены в коде теста | Использовать secret manager или env‑переменные |
| **Нарушение AAA (Arrange‑Act‑Assert)** | Нет чёткого разделения Arrange, Act, Assert | Разделить тест на три явных блока с комментариями или пустыми строками |
| **Отсутствие покрытия error paths** | Нет тестов на null, невалидные входные данные, исключения | Добавить тесты для каждого негативного сценария: `null`, `invalid`, `exception` |

### СРЕДНЯЯ важность (MEDIUM) — исправить в ближайшее время

| Test Smell | Симптом | Рекомендация |
|------------|---------|----------------|
| **Неверный тип assertion** | `Assert.Same` для `byte[]` вместо `Equal` | Использовать правильный тип проверки |
| **Перемокание (over‑mocking)** | Мокаются value‑объекты или простые POCO | Мокать только внешние зависимости |
| **Хрупкие тесты** | Ломаются при внутреннем рефакторинге | Тестировать поведение, а не реализацию |
| **Медленные тесты** | Юнит‑тест >100 мс | Оптимизировать или перенести в интеграционные |
| **Путаница мок/стаб** | Используется mock где нужен stub, или наоборот | Понимать test doubles (см. таблицу) |
| **Неспецифичное исключение** | `Assert.Throws<Exception>` вместо конкретного типа | Проверять конкретный тип и сообщение |

### НИЗКАЯ важность (LOW) — nice to have

| Test Smell | Симптом | Рекомендация |
|------------|---------|----------------|
| **Магические данные** | Жёстко заданные значения без пояснений | Использовать именованные константы или строители |
| **Рулетка assertion** | Много проверок без чёткого фокуса | Группировать в одно логическое утверждение |
| **Дублирование тестового кода** | Copy‑paste setup | Вынести в хелперы / базовые классы / методы расширения |
| **Отсутствие property‑based тестов** | Только примеры для широкого диапазона входных данных | Рассмотреть property‑based тесты (FsCheck, QuickCheck, Hypothesis) |

## Test Doubles (шпаргалка)

| Тип | Когда использовать | Пример |
|------|----------|---------|
| **Stub** | Возвращает фиксированный ответ, без проверки вызова | `repository.Get(id).Returns(fixedEntity)` |
| **Mock** | Проверить взаимодействие (метод вызван с нужными аргументами) | `mock.Verify(x => x.Save(...), Times.Once)` |
| **Fake** | Легковесная реализация (например, in‑memory БД) | `InMemoryDbContext` |
| **Spy** | Записывает вызовы для последующей проверки | Лог вызовов |

## Чеклист с приоритетами

### HIGH (обязательно)

- [ ] **AAA**: Arrange‑Act‑Assert чётко разделены?
- [ ] **Одна ответственность**: Один тест = одно поведение?
- [ ] **Независимость**: Нет общего состояния? Свежие фикстуры?
- [ ] **Внешние зависимости замоканы**: Никаких реальных БД/API в unit‑тестах?
- [ ] **У каждого теста есть assertions**, проверяющие **бизнес‑результат** (а не вызов метода)?
- [ ] **Покрыты негативные сценарии**: null, невалидные входные, исключения с конкретным типом?
- [ ] **Нет flaky паттернов**: `Thread.Sleep`, random, гонки async, общее mutable состояние?
- [ ] **Асинхронные тесты**: везде `await`, метод возвращает `Task`, нет `.Result`/`.Wait()`?
- [ ] **Секреты не хардкодятся**: Нет паролей, ключей, токенов в коде теста?

### MEDIUM (желательно)

- [ ] **Именование**: `{Метод}_{Сценарий}_ОжидаемыйРезультат` или стандарт проекта?
- [ ] **Нет магических данных**: константы или строители с понятными именами?
- [ ] **Правильные типы assertions** (`Equal`, `Same`, `NotNull` и т.д.)?
- [ ] **Mock‑верификация только когда важна логика вызова** (не перепроверять каждую мелочь)?
- [ ] **Test doubles соответствуют назначению** (mock vs stub vs fake)?
- [ ] **Покрытие** (если автоматизировано): ≥80% на изменённом коде?

### LOW (хорошо бы)

- [ ] **DRY**: Нет дублирования тестового кода; хелперы вынесены?
- [ ] **Читаемость**: Тест читается как спецификация?
- [ ] **Рассмотрены property‑based тесты** для многомерных входных данных?

## Формат вывода (только JSON)

Никакого дополнительного текста вне JSON.

```json
{
  "verdict": "APPROVED | CONDITIONALLY_APPROVED | REJECTED",
  "findings": [
    {
      "severity": "HIGH|MEDIUM|LOW",
      "category": "Test Structure|Naming|Data|Assertions|Mocks|Coverage|Maintainability|Security|Business Logic|Async|Flakiness",
      "line_start": 42,
      "line_end": 45,
      "problem": "Описание проблемы на русском",
      "suggestion": "Конкретное исправление (код или текст)"
    }
  ],
  "coverage": {
    "lineCoverage": 75.5,
    "branchCoverage": 68.2,
    "threshold": 80,
    "passed": false
  },
  "note": "Необязательное сообщение"
}
```

- `APPROVED` → нет HIGH находок.  
- `CONDITIONALLY_APPROVED` → есть MEDIUM (можно принять, но исправить в ближайшее время).  
- `REJECTED` → есть хотя бы одна HIGH находка (нельзя merge).  
- `line_start`/`line_end` необязательны (если неизвестны — 0).

## Автоматическая проверка покрытия (языконезависимо)

### Предварительные требования
- Фреймворк с coverage (например, `pytest-cov`, `jest --coverage`, `dotnet test --collect`, `mvn jacoco:report`).

### Алгоритм

1. **Найти изменённые файлы** (`git diff --name-only`) → отфильтровать исходные (исключить тесты).
2. **Запустить тесты с coverage** → получить отчёт (Cobertura, JaCoCo, lcov и т.п.).
3. **Разобрать отчёт** → построчная/ветвечная покрытие на файл/метод.
4. **Для файлов ниже порога (≥80%)** → вывести список непокрытых методов (не более 5).
5. **Сгенерировать заглушки тестов** для непокрытых методов.

### Пример отчёта покрытия (Cobertura XML)

```xml
<coverage line-rate="0.75" branch-rate="0.68">
  <packages>
    <package name="Example.Service">
      <classes>
        <class name="ExampleService" filename="Services/ExampleService.cs" line-rate="0.45">
          <methods>
            <method name="CreateRequest" line-rate="0.0">
              <lines><line number="45" hits="0" branch="true"/></lines>
            </method>
          </methods>
        </class>
      </classes>
    </package>
  </packages>
</coverage>
```

### Логика генерации заглушек тестов

Для каждого непокрытого метода (максимум 5):
- Прочитать код метода → извлечь тело.
- Проанализировать сигнатуру (параметры, возвращаемый тип, исключения).
- Посчитать ветвления (`if`, `switch`, проверки на null) → по одной заглушке на ветку.
- Сгенерировать заглушки для:
  - Счастливый путь (валидный ввод)
  - Null‑вход (если применимо)
  - Невалидный ввод (ошибка валидации)
  - Каждая явная ветка

**Пример (C#):**

```csharp
// Продуктивный код
public RequestResult CreateRequest(RequestDto request)
{
    if (request == null) throw new ArgumentNullException();
    if (!IsValid(request)) return RequestResult.Failed("Invalid");
    return RequestResult.Success(_repo.Create(request));
}

// Сгенерированные заглушки
[Fact] public void CreateRequest_Valid_ReturnsSuccess() { /* Arrange, Act, Assert */ }
[Fact] public void CreateRequest_Null_Throws() { /* ... */ }
[Fact] public void CreateRequest_Invalid_ReturnsFailed() { /* ... */ }
```

### Команды для разных языков

| Язык | Команда |
|----------|---------|
| .NET | `dotnet test --collect:"XPlat Code Coverage"` |
| Java (Maven) | `mvn test jacoco:report` |
| Python | `pytest --cov=src --cov-report=xml` |
| JavaScript (Jest) | `npm test -- --coverage` |

### Краевые случаи проверки покрытия

| Сценарий | Действие |
|----------|----------|
| Нет изменённых исходных файлов | `verdict: APPROVED`, пропустить coverage |
| Нет тестовых проектов | `verdict: ERROR`, сообщение в `note` |
| Отчёт о покрытии не найден | `verdict: ERROR`, сообщение в `note` |
| >5 непокрытых методов | Сгенерировать первые 5, остальные перечислить в `note` |
| Файл с 0% покрытия | Все методы непокрыты, сгенерировать тесты |

## Примеры границ интеграции с другими ролями

| Роль | Что проверяет | Пример замечания |
|------|----------------|------------------|
| **review-analyst** | Покрытие бизнес‑сценариев | «Не хватает случая с дублирующим email» |
| **review-architect** | Направление зависимостей | «Сервис не должен напрямую зависеть от DbContext» |
| **review-tester** | **Качество тестов** | «Assert проверяет вызов `Save()`, а не бизнес‑результат» |
| **review-security** | Корректность безопасности | «Пароль логируется в тесте» |

**Плохой тест (разные роли найдут проблемы):**

```csharp
[Fact]
public void ProcessPayment_ShouldWork()
{
    var mock = new Mock<IPaymentService>();
    mock.Setup(x => x.Charge(It.IsAny<decimal>())).Returns(true);
    var service = new PaymentProcessor(mock.Object, "test_key");
    var result = service.Process(100m);
    mock.Verify(x => x.Charge(100m), Times.Once);
}
```

- **review-tester** скажет: «Assert проверяет вызов метода, а не бизнес‑результат – надо проверить `result.Status == PaymentStatus.Success`»
- **review-security** скажет: «Хардкод API‑ключа – используй secret manager»

## Соответствие бизнес‑логике (примеры)

**Хорошо:**
- Мок возвращает реалистичный IBAN: `"DE89370400440532013000"`
- Assert проверяет бизнес‑результат: `result.Status == ApprovalStatus.Approved`
- Граничный случай из бизнеса: `amount > MaxTransactionLimit`

**Плохо:**
- Мок возвращает плейсхолдер: `"test_iban"`
- Assert проверяет реализацию: `repository.Verify(x => x.Save(), Times.Once)`
- Граничный случай не из бизнеса: `amount = -999999`

## Краткое руководство по использованию

1. Написать тест (или получить PR с тестами).
2. Запустить `review-tester` (self‑review или code review).
3. Если включена проверка покрытия → запустить автоматический анализ покрытия.
4. Вывести JSON → исправить HIGH находки немедленно, MEDIUM – в ближайшее время.
5. Закоммитить / вмержить.

## Юнит vs Интеграционные тесты (кратко)

| Критерий | Юнит‑тесты | Интеграционные тесты |
|----------|-----------|----------------------|
| Моки | Обязательны для внешних зависимостей | Минимум, реальные сервисы |
| Скорость | <100 мс каждый | Могут быть медленными |
| Изоляция | Полная изоляция | Изолированы от production, но не от инфраструктуры |
| Данные | In‑memory / моки | Тестовая БД / очереди |
| Assertions | Бизнес‑результат | End‑to‑end результат |

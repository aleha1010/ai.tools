Создай структурированный документ эскалации при блокировке работы над планом.

## Когда использовать

**Ручной вызов:**
- Пользователь упоминает "эскалация", "escalation", "проблема с планом", "заблокировано"
- Нет информации для продолжения работы
- Обнаружен блокер, который невозможно решить самостоятельно

**Автоматический вызов (в task_loop):**
- `current_rejections >= 5` для одной задачи (task-level rejection limit)

## Обработка ошибок (КРИТИЧНО)

| Ситуация | Действие |
|----------|----------|
| `plan_path` не найден | Создать minimal escalation: `severity=BLOCKER`, `problem_description="Plan not found: {path}"` |
| `tasks.md` не найден или пуст | `completed_tasks: [] (нет выполненных задач)`, `blocked_tasks: [] (нет задач)` |
| Перезапись `.escalation_handoff.md` | Сохранить предыдущую версию как `.escalation_handoff.prev.md` |

## Определение severity

| Ситуация | Severity |
|----------|----------|
| Нет критической информации, все reviewers REJECTED | BLOCKER |
| 3+/5 reviewers REJECTED, есть workaround | HIGH |
| 1-2 reviewers REJECTED, возможен обход | MEDIUM |
| Файл существует, но пустой | HIGH |
| 5+ НЕПРЕРЫВНЫХ REJECTED для одной задачи (task rejection limit) | BLOCKER |
| Mixed results (например 3 REJECTED + 2 APPROVED) | HIGH (требует анализа) |

## Интеграция argument-hint

- `problem_description` заполняется из аргумента пользователя при ручном вызове
- Если аргумент пуст → извлекается из контекста последней failed итерации

## Примеры конкретных решений

В секции "Требуемые решения" указывай конкретные действия:

- "Подтвердить выбор варианта A (использовать тестовую БД)"
- "Предоставить значение PROD_DATABASE_URL через переменные окружения"
- "Разрешить конфликт зависимостей между review-security и review-architect-backend"

## Обязательный шаблон .escalation_handoff.md

```markdown
# Escalation Handoff

**Plan:** `.kilo/plans/{plan-id}.md`
**Timestamp:** ISO-8601
**Severity:** BLOCKER | HIGH | MEDIUM

## Реализованные задачи

- [x] T001 — Название задачи
- [ ] T002 — Заблокирована (причина)
- [ ] T003 — Текущая задача (заблокирована)

## Обнаруженная проблема

[Описание проблемы — что именно заблокировано и почему]

## Контекст

[Дополнительная информация:
- Что было сделано
- Какие файлы читались
- Какие решения принимались
- Почему недостаточно текущей информации]

## Варианты решения

1. **Вариант A** — [описание]
   - Плюсы: ...
   - Минусы: ...
2. **Вариант B** — [описание]
   - Плюсы: ...
   - Минусы: ...

## Требуемые решения

- [ ] [решение 1 — что конкретно нужно от пользователя или planning agent]
- [ ] [решение 2]
```

## Алгоритм заполнения полей

| Поле | Источник | Приоритет |
|------|----------|-----------|
| `plan_path` | Контекст текущего плана | Валидировать существование файла |
| `completed_tasks` | `.kilo/plans/tasks.md` (задачи со статусом `[x]`) | Если файл не найден → `[]` |
| `blocked_tasks` | 1. Секция "Depends On" в tasks.md | Явные зависимости |
| | 2. Явное указание в описании задачи | Если нет секции Depends On |
| | 3. Пустой массив | Если зависимостей нет |
| `severity` | Таблица примеров выше | По ситуации |
| `problem_description` | argument-hint или контекст failed итерации | Приоритет: аргумент пользователя |

## Расположение файла

- Всегда: `.kilo/plans/.escalation_handoff.md`
- Перезаписывать существующий файл (одна активная эскалация на план)
- При перезаписи сохранить предыдущую версию: `.escalation_handoff.prev.md`

## Инструкция выполнения

1. Определить необходимость эскалации (ручной/автоматический триггер)
2. Собрать контекст:
   - Найти текущий план
   - Прочитать tasks.md для completed_tasks
   - Определить blocked_tasks по зависимостям
3. Определить severity по таблице
4. Сформировать problem_description:
   - Использовать argument-hint если передан
   - Иначе извлечь из контекста последней итерации
5. Если существует `.escalation_handoff.md`:
   - Сохранить как `.escalation_handoff.prev.md`
6. Создать новый `.escalation_handoff.md` по шаблону
7. Вывести пользователю краткое резюме эскалации
